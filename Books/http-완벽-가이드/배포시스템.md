📘 HTTP 완벽 가이드 — 19장 핵심 정리

# 1. 19장의 핵심 주제

19장은 다음 두 가지를 다루는 장이다: 1. 웹 콘텐츠의 배포(Deploy)와 공동 편집(Co-authoring) 문제를
HTTP 확장으로 해결하려고 시도한 역사 2. HTTP가 읽기(GET) 중심에서 쓰기/편집 기능을 갖추려는 과정

오늘날에는 Git·CI/CD·Cloud Storage가 이 역할을 맡지만,
HTTP 자체에 쓰기 기능을 직접 포함하려 했던 시도가 바로 19장의 내용이다.

---

# 2. 왜 이 장이 중요할까? (현대 개발과의 연결)

19장은 `“옛날 기술”`처럼 보이지만,
사실 아래 개념들의 뿌리를 이해하게 해준다:
• 왜 PUT, DELETE, COPY, MOVE, LOCK 같은 메서드가 존재하는가?
• REST API의 리소스 중심 구조가 어떤 배경에서 나왔는가?
• 왜 현대에는 WebDAV 대신 Git/CI/CD가 사용되는가?
• S3 같은 오브젝트 스토리지가 왜 PUT 기반인지?
• 웹에서 파일 시스템처럼 조작하려 했던 시도가 있었음을 이해

즉,
19장은 REST·Git·S3·DevOps의 올라운드 역사 배경이다.

---

# 3. FrontPage Server Extensions (19.1)

웹을 통해 직접 웹사이트를 만들고 배포하려던 초기 시도.

핵심 기능
• 웹 페이지를 직접 서버에 수정/업로드
• 파일 잠금(lock)
• 파일 충돌 방지
• 디렉터리 생성/삭제

한계
• 협업 충돌 해결이 약함
• 버전 관리 없음
• 직접 서버 파일 조작 → 위험
• 확장성 부족

➡ 이러한 한계 때문에 결국 Git + CI/CD 시대로 이동.

---

# 4. WebDAV (19.2 핵심)

`WebDAV = HTTP를 파일 시스템처럼 사용하기 위한 확장`

즉:
`“웹 브라우저나 툴이 웹 서버를 마치 폴더처럼 다루게 하자.”`

WebDAV가 추가한 메서드

메서드 의미
PUT 파일 업로드/대체
DELETE 파일 삭제
COPY, MOVE 파일 이동/복사
MKCOL 디렉터리(컬렉션) 생성
PROPFIND 메타데이터 조회
LOCK / UNLOCK 파일 잠금(편집 충돌 방지)

이 메서드들은 “웹을 파일 시스템화”하기 위한 조작들이다.

---

# 5. WebDAV가 REST API 설계에 준 영향 (중요)

WebDAV의 시도는 REST의 개념적 기초가 되었다.

1. PUT/DELETE = REST의 리소스 조작 원형

WebDAV → PUT을 “리소스를 전체 대체” 의미로 사용
REST → 같은 의미 유지

2. LOCK → REST의 봉쇠 정책

WebDAV = 잠금 기반 충돌 회피
REST = ETag / If-Match 기반 Optimistic Locking

3. PROPFIND → REST의 메타데이터 조회 개념

WebDAV가 구조적 리소스 설명의 필요성을 드러냄
→ REST / GraphQL에서 Representation 개념으로 발전

4. 컬렉션(MKCOL) → REST의 Collection/Item 모델과 동일한 철학

/users vs /users/{id}의 구조적 배경

---

# 6. 왜 WebDAV에서 POST가 거의 등장하지 않는가?

WebDAV는 “파일 시스템”을 HTTP로 구현하려는 시도였다.
그러므로 대부분의 작업은 정형화된 CRUD로 충분했다.
• 파일 만들기/업로드 = PUT
• 삭제 = DELETE
• 폴더 생성 = MKCOL
• 이동 = MOVE
• 속성 조회 = PROPFIND

POST는 `“서버가 알아서 처리하는 행동(action)”`이므로
파일 시스템 모델과 맞지 않음.

➡ REST 시대에 들어서 POST가 주요 메서드로 자리 잡은 것.

---

# 7. WebDAV의 한계 → Git/CI/CD로 이어짐

WebDAV의 한계 현대 기술의 해결
잠금(lock) 방식 협업 Git의 브랜치/머지
충돌 해결 취약 Pull Request, Review
수동 파일 업로드 CI/CD 자동화
웹 서버 파일 직접 수정 GitOps / 배포 자동화
리소스 중심 CRUD 제한 REST/GraphQL 확장
단일 서버 의존 분산형 Git

결국 “웹 서버 자체를 파일 시스템처럼 다루는 방식”은
협업·안전성·버전 관리 등에서 제한이 많았고,
대규모 소프트웨어 개발에서는 Git이 승리하게 됨.

---

# 8. WebDAV 기능은 오늘날 어디에서 이어졌는가?

놀랍게도 WebDAV의 철학은 Object Storage에서 그대로 살아 있음.

WebDAV S3/GCS/Azure Blob
PUT PUT Object
DELETE DELETE Object
PROPFIND HEAD Object (메타데이터 조회)
COPY CopyObject
MKCOL Prefix/폴더 개념 (가상)

즉:

WebDAV의 “파일시스템 같은 HTTP” 철학은
클라우드 스토리지에서 현대적으로 계승됨.

---

# 9. 19장의 진짜 교훈 (현대 개발자 기준)

19장은 이렇게 말하고 있음:

“HTTP가 문서 조회 시스템을 넘어서
배포·편집·협업을 담당하려 한 시도가 있었지만,
그것이 Git/CI/CD/Cloud 시대의 뿌리가 되었다.”

즉,

✔ WebDAV = HTTP가 write를 담당하려던 시대

✔ Git/CI/CD = write/배포가 HTTP 밖으로 빠져나온 시대

✔ Object Storage = WebDAV의 기능을 현대화한 형태

이 관계를 이해하면
HTTP 메서드 구조 + REST 철학 + Git/DevOps의 태동까지 모두 이어짐.

---

# 10. 19장 전체를 한 문단으로 요약

HTTP는 처음엔 문서 읽기(GET) 전용이었다.
하지만 웹 콘텐츠를 수정·배포하고 공동 편집할 필요가 생기면서
FrontPage, WebDAV 같은 HTTP 확장들이 등장했다.
이것은 PUT/DELETE/COPY/MOVE/LOCK 등 파일시스템적 메서드들을 만들었고
REST의 리소스 조작 개념에 영향을 주었으며,
현대에는 Git, CI/CD, Object Storage로 철학이 계승되었다.
